[[_api_mqtt]]
= API MQTT

Publishes OpenEMS Edge data to a MQTT broker and subscribes to control topics.
Integrates OpenEMS with IoT platforms, cloud dashboards, and home automation systems via MQTT protocol.

https://github.com/OpenEMS/openems/tree/develop/io.openems.edge.controller.api.mqtt[Source Code icon:github[]]

== Overview

The MQTT API Controller publishes OpenEMS channel values to a MQTT broker in real-time and exposes the Edge configuration.
This enables:

* **IoT Integration**: Connect to platforms like Home Assistant, Node-RED, Mosquitto dashboards
* **Cloud Connectivity**: Publish to Azure IoT Hub, AWS IoT Core, Alibaba Cloud, etc.
* **Flexible Topic Filtering**: Selective channel publication via MQTT topic subscriptions
* **Real-Time Monitoring**: Changes published immediately; periodic updates every 5 minutes
* **Configuration Export**: Full Edge config published as JSON for remote discovery
* **TLS/SSL Support**: Encrypted connections with certificate authentication

.Key Capabilities
* Automatic reconnection with exponential backoff
* Change-based and periodic publishing
* Topic filtering with MQTT subscription patterns
* Persistence priority filtering
* Debug logging mode
* Certificate-based authentication

|===

|===

[[_configuration_mqtt_filtering]]
=== Topic Filtering

[cols="1,3"]
|===
| Parameter | Description

| `topicFilters` | Array of MQTT topic subscription patterns
|===

Uses standard MQTT wildcarding:
* `+` matches single level: `channel/_sum/+` matches `channel/_sum/GridActivePower`
* `#` matches multiple levels: `channel/_sum/#` matches `channel/_sum/GridActivePower`, `channel/_sum/EssSoc`, etc.

If not set, all channels are published.

.Example Filters
[source]
----
topicFilters = [
  "channel/lastUpdate",
  "channel/_sum/#",
  "channel/ess0/+",
  "edgeConfig"
]
----

== Topic Structure

The MQTT topic hierarchy is:

[source]
----
[topicPrefix/]edge/{clientId}/channel/{componentId}/{channelId}
[topicPrefix/]edge/{clientId}/channel/lastUpdate
[topicPrefix/]edge/{clientId}/edgeConfig
----

=== Examples

.Basic Topics (no prefix)
[cols="1,2"]
|===
| Topic | Content

| `edge/edge0/channel/_sum/GridActivePower` | Sum grid power (W)
| `edge/edge0/channel/ess0/Soc` | ESS state of charge (%)
| `edge/edge0/channel/meter0/ActivePower` | Meter power reading (W)
| `edge/edge0/channel/lastUpdate` | Unix timestamp of latest update
| `edge/edge0/edgeConfig` | Full Edge configuration as JSON
|===

.With Topic Prefix
If `topicPrefix` is set to `factory1`:

[cols="1,2"]
|===
| Topic | Content

| `factory1/edge/edge0/channel/_sum/GridActivePower` | Sum grid power
| `factory1/edge/edge0/channel/lastUpdate` | Latest update timestamp
| `factory1/edge/edge0/edgeConfig` | Full Edge configuration
|===

== Publishing Behavior

=== Channel Data Topics

* Published when channel value changes
* Also published periodically every 5 minutes (keep-alive)
* Only channels matching `persistencePriority` threshold are published
* JSON payload format: value or object depending on channel type

.Example Channel Payload
[source,json]
----
{
  "GridActivePower": 2500,
  "EssSoc": 85.5,
  "GridFrequency": 50.0
}
----

=== Last Update Topic

Published alongside channel updates:

[source]
----
edge/edge0/channel/lastUpdate: 1711611234567
----

Unix timestamp in milliseconds. Useful for detecting stale data.

=== Edge Configuration Topic

Published at startup and whenever Edge configuration changes:

[source,json]
----
{
  "components": [
    {
      "id": "_sum",
      "alias": "Summary",
      "channels": [...]
    },
    {
      "id": "ess0",
      "alias": "Battery System",
      "channels": [...]
    }
  ]
}
----

== Persistence Priority Levels

Controls which channels are published based on their configured priority:

[cols="1,2"]
|===
| Priority | Channel Types

| `VERY_LOW` | All channels published (default)
| `LOW` | Excludes diagnostic channels
| `MEDIUM` | Only primary control/monitoring channels
| `HIGH` | Only critical channels (power, SOC)
| `VERY_HIGH` | System-critical only
|===

Lower `persistencePriority` threshold = more data published.

== Reconnection Strategy

Automatic reconnection with exponential backoff:

1. Initial reconnect delay: 5 seconds
2. Multiplier per attempt: 1.5x
3. Maximum delay: 5 minutes (300 seconds)

Connection failures are logged; Edge continues operating independently.

== Error Handling

[cols="1,2"]
|===
| Condition | Handling

| Connection failed | Automatic reconnection with backoff
| TLS certificate invalid | Connection rejected, error logged, retry scheduled
| Authentication failed | Connection rejected, check credentials
| Topic subscription rejected | Logged as warning; continues publishing
| Message publish timeout | Logged; next cycle attempts retry
| Configuration error | Component fails to activate, check syntax
|===

== Security Considerations

=== Transport Security

* **tcp://** - Unencrypted (use only on trusted local networks)
* **ssl://** and **mqtts://** - TLS encrypted (recommended)
* **ws://** - WebSocket unencrypted
* **wss://** - WebSocket encrypted (recommended for cloud)

=== Authentication

* Username/password: Basic MQTT authentication
* Certificates: Certificate-based mutual TLS authentication
* Token-based: Some brokers support auth tokens in username field (Azure, AWS)

=== Best Practices

1. Always use TLS/SSL for production and cloud connections
2. Use strong passwords or certificate authentication
3. Limit topic filters to necessary channels (reduce exposure)
4. Use separate client IDs for development vs production
5. Restrict broker firewall access to Edge IP addresses

== Common Integration Patterns

=== Home Assistant

1. Configure with MQTT broker (Mosquitto or built-in)
2. Set `topicPrefix` to match Home Assistant expectations
3. Create MQTT sensor entities pointing to Edge topics
4. Enable automations for control (write via Home Assistant MQTT integration)

=== Node-RED

1. Add MQTT nodes with broker connection details
2. Subscribe to `edge/+/channel/#` for all data
3. Process and forward to databases, visualizations, or other systems

=== Prometheus + Grafana

1. Use MQTT-to-Prometheus bridge (e.g., mqtt-exporter)
2. Configure with appropriate topic filters
3. Scrape metrics and visualize in Grafana

== Troubleshooting

.Cannot Connect to Broker
Check URI format (tcp:// vs ssl://), firewall rules, broker listening on configured port, credentials.

.Topics Not Appearing
Verify `topicFilters` patterns are correct, `persistencePriority` is not filtering channels, debug mode shows publishing attempts.

.TLS Connection Fails
Validate certificate chain (certPem + trustStorePem), check certificate validity dates, ensure certificate matches broker domain.

.Stale Data Detection
Subscribe to `channel/lastUpdate` topic; if timestamp hasn't updated in reasonable time, connection may be degraded.
