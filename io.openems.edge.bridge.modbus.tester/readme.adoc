= Bridge Modbus Tester

A minimal Modbus device component for testing and debugging a physical Modbus communication channel. It reads a configurable register via FC3 (Read Holding Registers) and exposes raw response data on channels, enabling verification that hardware-level Modbus communication works through the modbus bridge without needing an implemented device-specific OpenEMS component.

== Channels

[cols="1,1,3"]
|===
| Channel | Type | Description

| `Response` | STRING | Raw register payload as hex (e.g. `003F 00A1`)
| `Message` | STRING | Full reconstructed Modbus frame (protocol-dependent, for comparison with bridge logs)
| `RegisterAddress` | INTEGER | Configured register start address
| `RegisterCount` | INTEGER | Number of registers being read
| `UnitId` | INTEGER | Configured Modbus slave address
| `ModbusProtocol` | STRING | Configured transport protocol (TCP/RTU/ASCII)
| `CommunicationOk` | BOOLEAN | `true` after successful read, `false` after error
| `CommunicationErrors` | INTEGER | Running error counter
| `LastError` | STRING | Last error message text
|===

== Configuring the Modbus Bridge for deep debugging

The bridge's `LogVerbosity` configuration controls what the bridge itself logs. Available levels (from `io.openems.edge.bridge.modbus.api.LogVerbosity`):

[cols="1,3"]
|===
| Level | What it shows

| `NONE` | No logs
| `DEBUG_LOG` | Basic info in Controller.Debug.Log
| `READS_AND_WRITES` | All read/write requests
| `READS_AND_WRITES_VERBOSE` | Reads/writes + hex values of request and response
| `READS_AND_WRITES_DURATION` | Reads/writes + timing per request
| `READS_AND_WRITES_DURATION_TRACE_EVENTS` | All of the above + internal state machine trace
|===

*Recommended for debugging:* Set the bridge's `logVerbosity` to `READS_AND_WRITES_VERBOSE` or `READS_AND_WRITES_DURATION_TRACE_EVENTS`. This gives wire-level hex frames from the bridge side, which can be compared with the tester's `Message` channel to cross-validate.

== Configuring the tester component

Example configuration (Apache Felix Web Console or JSON):

----
id:                modbusTester0
modbus_id:         modbusAscii0        # must match the bridge component ID
modbusUnitId:      1                   # slave address of the target device
registerAddress:   1                   # register to read (0-based)
registerCount:     2                   # number of consecutive 16-bit registers
modbusProtocolType: ASCII              # must match the bridge type for correct Message reconstruction
----

== Debugging workflow

1. Configure and start the Modbus bridge (e.g. `modbusAscii0`) with `logVerbosity = READS_AND_WRITES_VERBOSE`
2. Configure and start `modbusTester0` pointing to that bridge
3. Observe:
   - *Bridge logs*: Raw Modbus frames at transport level
   - *Tester `Response` channel*: Hex payload of the register data
   - *Tester `Message` channel*: Reconstructed full Modbus frame (for comparison with bridge logs)
   - *Tester `CommunicationOk` channel*: Whether reads succeed
   - *Tester `LastError` channel*: Error details on failure

== Note on the ASCII bridge's transport listener

The ASCII bridge implementation (`BridgeModbusSerialAsciiImpl`) includes an `AsciiTransportListener` that tracks:

- `bytesSent` / `bytesReceived`
- `communicationErrors` / `successfulTransactions`
- `lastSuccessfulCommunication` timestamp

These are available as channels on the bridge component itself and complement the tester's per-register view.
